# Lab 1: Wireshark Traffic Analysis

## Objective
Capture live network traffic, analyze core protocols (ARP, ICMP, DNS, HTTP, TLS), 
and practice SOC-relevant traffic analysis techniques including device identification, 
OS fingerprinting, and encrypted traffic inspection.

## Tools Used
- Wireshark 4.6.4 (macOS — GUI packet analyzer)
- curl (HTTP request generation)
- ping (ICMP traffic generation)
- nslookup (DNS query generation)

## Date
March 12, 2026

## Lab Environment
- **Capture Point:** MacBook Pro (192.168.178.111) — en0 interface (Wi-Fi)
- **Network:** 192.168.178.0/24
- **Gateway:** Fritz!Box (192.168.178.1)
- **Capture Duration:** ~2 minutes
- **Total Packets Captured:** 2,173

```

```

## Phase 1: ICMP Analysis

**Filter:** `icmp`

### Local Network Ping (Fritz!Box + iMac)
- 16 ICMP packets captured (4 request + 4 reply × 2 targets)
- TTL = 64 for all local packets — no routers between devices
- 0% packet loss — both devices healthy and reachable

### External Ping (google.com → 142.251.140.174)
- TTL on request (MacBook → Google): **64** — MacBook's starting TTL
- TTL on reply (Google → MacBook): **119** — started at 128, crossed 9 hops
- 128 - 119 = **9 routers** between MacBook and Google

### Key Takeaway: OS Fingerprinting via TTL
| Starting TTL | Typical OS |
|:---:|:---|
| 64 | Linux / macOS |
| 128 | Windows |
| 255 | Network devices (routers, switches) |

Observing the reply TTL allows estimation of both hop count and remote OS — 
a passive reconnaissance technique relevant to both Net+ and Security+.

### Key Takeaway: NAT Visibility
Wireshark captures traffic **before NAT translation** at the Fritz!Box. 
Outbound packets show the private IP (192.168.178.111), not the public IP. 
The capture point determines what you see — a critical concept for SOC analysts 
deciding where to place network sensors.

**Filter:** `dns`

### Background DNS Traffic (Not Generated by Me)
Applications running in the background produced constant DNS queries:

| Domain | Application |
|:-------|:-----------|
| spclient.wg.spotify.com | Spotify client |
| seed-mix-image.spotifycdn.com | Spotify CDN |
| api.github.com | GitHub |
| collector.github.com | GitHub telemetry |

### Key Takeaway
SOC analysts must distinguish between **user-generated** and **background application** 
DNS traffic. Unexpected or suspicious domain queries in DNS logs are a primary indicator 
of malware C2 communication.

## Phase 3: HTTP Analysis

**Filter:** `http`

### TCP 3-Way Handshake (example.com — 104.18.27.120)

| Packet | Direction | Flag | Meaning |
|:-------|:----------|:-----|:--------|
| 2059 | MacBook → example.com | [SYN] | Connection request |
| 2060 | example.com → MacBook | [SYN, ACK] | Request accepted |
| 2061 | MacBook → example.com | [ACK] | Connection established |

### SYN Packet Details
| Field | Value | Meaning |
|:------|:------|:--------|
| 50552 → 80 | Ports | Ephemeral port → HTTP well-known port |
| Seq=0 | Sequence | First packet in stream |
| Win=65535 | Window Size | Flow control — max receivable bytes |
| MSS=1460 | Max Segment Size | Max payload per packet |
| SACK_PERM | Selective ACK | Supports selective acknowledgment |

### HTTP Request (Packet 2062)
```
HEAD / HTTP/1.1
Host: example.com
User-Agent: curl/8.7.1
Accept: */*
```
**SOC Note:** User-Agent reveals the tool used. `curl` in server logs may indicate 
automated scanning or reconnaissance. SOC analysts monitor for non-browser User-Agents 
(curl, wget, python-requests, nikto, sqlmap) as potential threat indicators.

### HTTP Response (Packet 2064)
```
HTTP/1.1 200 OK
Server: cloudflare
CF-RAY: 9db24be5cae5e521-TXL
Content-Type: text/html
cf-cache-status: HIT
```
**Analysis:**
- **Server: cloudflare** — reveals infrastructure (information leakage)
- **CF-RAY: TXL** — response served from Berlin datacenter (geolocation indicator)
- **cf-cache-status: HIT** — content served from cache, not origin server
- Response time: 24.958ms

### TCP Connection Teardown

| Packet | Flag | Meaning |
|:-------|:-----|:--------|
| 2066 | MacBook → [FIN, ACK] | "I'm done, closing" |
| 2067 | example.com → [FIN, ACK] | "Me too, closing" |
| 2068 | MacBook → [ACK] | "Confirmed, goodbye" |

Full TCP lifecycle observed: **Handshake → Data Transfer → Teardown** (10 packets total)

## Phase 4: ARP Analysis

**Filter:** `arp`

### Normal ARP Behavior
| Request | Reply |
|:--------|:------|
| "Who has 192.168.178.111? Tell 192.168.178.1" | "192.168.178.111 is at 7a:7d:0f:d3:1b:78" |
| "Who has 192.168.178.78? Tell 192.168.178.111" | "192.168.178.78 is at 9a:03:8e:02:c8:dc" |

iMac's ARP reply shows the repeater's MAC — consistent with Lab 0 findings.

### Suspicious Pattern Detected
Repeated unanswered ARP requests observed:
```
"Who has 192.168.178.65? Tell 192.168.178.97"
```
This request appeared **15+ times** throughout the capture with **zero replies**.

**Analysis:**
- Source: Amazon Fire Stick (192.168.178.97) — identified via HTTP traffic in this lab
- Target: 192.168.178.65 — **no device exists at this IP**
- Likely cause: Fire Stick searching for a previously paired device
- Risk assessment: Low (normal device behavior), but in a production environment 
  repeated ARP requests for non-existent hosts could indicate network reconnaissance

## Phase 5: TLS / Encrypted Traffic Analysis

**Filter:** `tls.handshake`

### Two Encryption Protocols Observed

**TLSv1.3 (Standard HTTPS):**

| Step | Message | Purpose |
|:-----|:--------|:--------|
| 1 | Client Hello | Client announces supported ciphers + target hostname (SNI) |
| 2 | Server Hello + Change Cipher Spec | Server selects cipher, encryption begins |

**QUIC (Google's UDP-based protocol):**
- Used by Spotify and Google services
- Faster than TCP+TLS (zero round-trip handshake)
- Fully encrypted — SNI may be hidden

### SNI (Server Name Indication) — Key SOC Technique
Even though HTTPS content is encrypted, the **SNI field in Client Hello** reveals 
the destination domain:

| SNI | Service |
|:----|:--------|
| api.github.com | GitHub API |
| collector.github.com | GitHub telemetry |
| seed-mix-image.spotifycdn.com | Spotify CDN |
| spclient.wg.spotify.com | Spotify client |
| i.scdn.co | Spotify content |

**SOC Application:** SNI-based monitoring allows threat detection in encrypted traffic 
without decryption. Malicious C2 domains in SNI fields trigger alerts in production SOCs.

This technique — identifying devices by their network behavior rather than active scanning — 
is a core SOC and threat hunting skill.

## Wireshark Filters Reference

| Filter | Purpose |
|:-------|:--------|
| `icmp` | Ping traffic |
| `dns` | DNS queries and responses |
| `http` | Unencrypted web traffic |
| `arp` | MAC address resolution |
| `tls.handshake` | TLS/SSL connection setup |
| `tcp.flags.syn == 1` | TCP connection initiations |
| `ip.addr == X.X.X.X` | All traffic to/from specific IP |
